Software analysis

1

1.1

PL即编程语言(programming language)，分为理论、环境与应用部分。支持语言运行的一个系统叫做环境(environment)。保证系统的可靠性与安全性指向语言的应用层面(application)，我们的程序分析就是来自这一层面。

语言主要分为三类，命令式编程语言(将语言拆解为一条一条来执行,c、java)、函数式编程语言(py、js)、逻辑式编程语言(通过逻辑的与或非声明来得到结果)，这个分类是不变的，但是编写的程序越来越复杂。那么这种复杂的程序/软件的安全或者其他问题如何解决？

为什么需要静态分析：

  • 发现潜在问题

    • 可靠性的保证：内存泄漏...
    • 安全性的帮助：私有信息的泄露，注入...

  • 编译优化：死代码

  • 理解程序

静态分析：想在运行程序P之前，我就获取程序的相关的行为，并且判断其是否满足某些性质。

  • 是否存在私有信息的泄露？
  • 是否存在空指针？
  • 会有指向相同内存地址的指针吗？
  • 是否存在死代码，可以移除以加速？
  • ...

莱斯定理：并不存在方法能准确判断程序是否满足某些性质(注意准确，即说有就有，说没有就没有)。换而言之，递归可枚举的语言(图灵机可识别的语言)的不那么简单(只要跟程序运行行为相关的，令人感兴趣的，我们都认为是有用的、不那么简单的)的性质，是无法准确判断(decidable)的(不能说是就是，说否就否)。

总结：不存在perfect的静态分析方法，来准确判断所有的性质。

perfect的静态分析满足：sound且complete

truth就是事实，比方说程序里存在10个空指针引用，这就是truth。我们不存在perfect的方法判断确实有10个，sound代表过近似(overapproximate)，我判断它有10+个，这可能是11，12个，也可能成千上万个，但是其中就是包含这10个空指针引用；complete代表欠近似(underapproximate)，我判断它有10-个，这可能是9，8个，爆出来的就是truth，但是不一定爆得全。

但是我们可以追求"useful"的静态分析，只需满足compromise soundness(false negative)或compromise completeness(false positive)中的一个。

妥协什么就是只追求另一个，妥协soundness就是只追求complete，这造成了漏报(false negative)；妥协completeness就是只追求sound，这造成误报(false positive)。

大部分的静态分析是妥协completeness的，即宁愿误报也不漏报(sound but not fully precise)。

上面的一个例子陈述了sound(理解为全面性似乎比较恰当)的重要性。如果你不sound，只检查了蓝色的数据流，那么如果有绿色数据流就会导致强制类型转换，可能出现运行时错误。不检查绿色数据流就是不sound的，反之是sound的，这两者最后的判断也产生了不同(safe cast or not)。

所以我们可以重新阐述静态分析的概念，即在保证或接近(ensure or get close to)soundness，在精度与开销(速度)上做一个平衡。

2

Intermediate Representation(IR):中间表示。

我们认为利于静态分析的IR是简便的，且能够表示程序的所有行为。

2.1

编译器的一个流程。

第一步骤：词法分析(Lexical Analysis)，将源代码转化为Tokens(使用正则表达式regular expression)。就拿英文举例子，You are a man就会被转换为4个合法的tokens，而You goouojd由于并不能组成合法的单词，因此就无法通过词法分析。

第二步骤：语法分析(Syntax Analysis)，将Tokens转化为AST(抽象语法树)(使用上下文无关文法 context-free grammar)。比如I am clever就能通过，那比如Like your hair I不符合语法规则，也就不通过。

第三步骤：语义分析(Semantic Analysis)，将AST转化为decorated AST(使用arribute grammar)。比如Apples eat you就不通过语义分析。

第四步骤：IR生成，将decorated AST转化为IR。

第五步骤：机器码转换，将IR转化为机器码。

显然是不会直接跳过前面合法性的检查来直接转换为IR的，因为如果前面基本的都通不过，那还要交给错误检测吗？

2.2

AST vs IR

AST:

  • high-level，接近源代码的
  • 与上一条相对应的，依赖于不同的语言
  • 利于快速类型检测
  • 看不出控制流的信息

IR:

  • low-level，接近机器码的

  • 与上一条对应，与语言依赖性不高(语言无关的)

    • 由于这一性质，IR成为静态分析的重头戏
    • 较简洁
    • 看得出控制流的信息

三地址码(3-Address Code 3AC)：

  • 在一个操作中，右侧最多只有一个操作符
  • 每个3AC最多含有3个地址

这个"地址"定义比较不一样，可以是：

  • name:a,b
  • constant:3
  • compiler-generated temporary:t1,t2

每一个操作都有其三地址码。

2.4

Static Single Assignment(SSA)

很好理解，就是每个变量只被赋值一次，比方说p、q这两个变量，它们会分出p1、p2、q1、q2...来表示同一个变量的不同状态。

但是这里，我们说每一个变量只被赋值一次，我们重新用其他变量名来表示不同状态。

为了避免歧义，SSA引入了"phi-function"\(\phi\)以应对控制流的交界处合并。

2.5

Control Flow Analysis：我们如何建立Control Flow Graph(CFG)？

CFG给予静态分析的基本结构，它的每一个节点可以是单独的3AC，也可以是多个3AC组成的Basic Block(BB)。

Basic Block(BB): 满足下列性质的最多的连续3AC序列。

  • 只有一个入口，且为第一条指令
  • 只有一个出口，且为最后一条指令

ok这里老师引导我们思考如何从3AC序列转换至BBs，我的思路就是贪心捕捉，第一条指令肯定就是第一个BB的开头，每一个BB持续拿指令，直到要么有第二个入口或者出现第二个出口我们就结束。来我们看看老师怎么说。

哦老师说的大差不差，就是第二个入口会是什么？无非就是其他指令跳转到的地方嘛，同理第二个出口就只会是要跳转到其他地方的指令。

leader是什么？很好理解的，跳转指令的下一条指令与其跳转到的地方就是所有的leader，当然整个3AC的总开头和总结尾也都是。

那么CFG的绘制就很简单了，BB到BB有边当且仅当：

  • BB1到BB2有有/无条件的跳转
  • BB2紧跟BB1，且BB1没有有/无条件跳转

BB1是BB2的前驱(predecessor)，BB2是BB1的后继(successor)。

还有两个特殊节点Entry与Exit，Entry是所有BB的前驱，Exit是所有BB的后继，但它们并不是可执行的IR。

Data Flow Analysis -- Applications

how applicaton-specific Data Flows through the :

  • Nodes(BBs/statements)：transfer function
  • Edges(control flow)：control-flow handling

回顾一下判断数字正负的例子，这里不具体说了。

may analysis:最常见的静态分析，输出可能是正确的(over-approximation)

must analysis: 输出一定是正确的(under-approximation)

不论是may还是must analysis，其目的都是保证分析的safety。

3.1

一个基本知识的概览。

Input and Output States:

  • 每一个IR statement的执行意味着从一个input状态转为output状态
  • input状态与program point before(after) the statement相关

注意这个^符号表示归于。

domain就是所有的可能抽象值的集合。

因此我们对数据流分析给出一个定义：

我们最常见的transfer function是forward analysis，\(OUT[s] = f_s(IN[s])\)，反过来的backward analysis，\(IN[s] = f_s(OUT[s])\)。

然后讨论控制流，有两种：

  • BB内部的，这显而易见是\(IN[s_{i+1}] = OUT[s_i]\)
  • BB之间的，这就要看BB之间的跳转了。明确:\(IN[B] = IN[s_1]\)和\(OUT[B] = OUT[s_n]\)，并且\(OUT[B] = f_B(IN[B]) \ \ f_B = f_{s_n} * ... * f_{s_1}\)和\(IN[B] = \cap P_{\ \ P \ a \ predecessors}\)。

3.2

Reaching Definition:

A definition \(d\) of a variable \(v\) is said to reach a program point \(p\) if there is a path from the entry of the procedure to \(p\) such that \(d\) is not killed along that path.

我们不妨翻译一下它，非常简单，就是说p reach q路径上如果没有新的定义(kill)。

这是可以用来检测的，比如对于变量初始化的检测，在程序入口增设循环定义，那如果一个变量\(v\)reach a point \(p\) where \(v\) is used，那么这个变量可能没被初始化。

这是一个May Analysis，老师的解释是，我们可能有成千上万条路径，每个可能被kill，也可能不，但是只要发现了一条kill就报错。

Live Variable Analysis：Live variables analysis tells whether the value of veariable v at program point p could be used along some path in CFG starting at p.If so , v is said to be live at p;otherwise, v is said to be dead at p.

就是一条路径上被使用且没有被重定义。

举个例子，寄存器的分配，假设一个情景，寄存器全满了但是我们想要用一个来存取值，我们最好使用the register with a dead value.

类似的抽象方式，之前是针对"definition"，现在针对"variable"。

老师提到使用backward analysis，因为这样可以自动向上传是否被use的信息，而不用像forward analysis那样每向下一个要与上一个关联。

与reaching definition类似，这是一个may analysis。
